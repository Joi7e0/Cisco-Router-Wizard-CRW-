# Процедура оновлення (Update / Maintenance Procedure)

Цей документ містить покрокові рекомендації для Release Engineer / DevOps щодо планового оновлення додатку "Cisco Router Wizard" у виробничому (production) середовищі. Зважаючи на архітектуру (локальний GUI на базі Python Eel), процес оновлення зводиться переважно до заміни виконуваного `.exe` файлу на робочих станціях або термінальних серверах (Terminal Service/RDS).

## 1. Підготовка до оновлення

### Створення резервних копій
Перед початком оновлення обов'язково збережіть поточну стабільну версію:
- Зробіть копію поточного виконуваного файлу `CiscoRouterWizard.exe` (наприклад, перейменуйте на `CiscoRouterWizard_vX.Y.Z_backup.exe`).
- Якщо з застосунком постачаються зовнішні конфігурації (наприклад, файл налаштувань користувача або окрема папка із шаблонами Jinja2, якщо вони не запаковані у білд), скопіюйте їх у безпечну директорію.

### Перевірка сумісності
- Переконайтеся, що новий Release Build скомпільований під цільову операційну систему (Windows 10/11, Windows Server 2016+).
- Перевірте Release Notes у репозиторії, щоб упевнитися, що нова версія не вимагає вищої версії вбудованого браузера WebView2/Chrome або зміни базових прав доступу.

### Планування часу простою (Downtime)
- Оскільки додаток є локальним і ізольованим, "час простою" стосується виключно інженерів, які в даний момент працюють з програмою на спільному сервері.
- Заплануйте оновлення у технологічне вікно (Maintenance Window).
- Завчасно (наприклад, за 15 хвилин) повідомте користувачів про необхідність згенерувати та зберегти поточні конфігурації у власні файли та закрити додаток.

## 2. Процес оновлення

### Зупинка потрібних служб
Оскільки це не фоновий сервіс (Windows Service), а десктопний UI-застосунок, його потрібно просто закрити.
- Попросіть користувачів закрити вікна `CiscoRouterWizard`.
- На Terminal Server (якщо застосовно) перевірте відсутність завислих процесів `CiscoRouterWizard.exe` у всіх сесіях. За потреби завершіть їх примусово (наприклад, через PowerShell: `Stop-Process -Name "CiscoRouterWizard" -Force`).

### Розгортання нового коду
- Скопіюйте новий скомпільований файл `CiscoRouterWizard.exe` з останнього релізу.
- Замініть старий виконуваний файл у розшареній або загальній директорії (наприклад, `C:\Program Files\CiscoRouterWizard\`).

### Міграція даних
- **Пропускається.** Застосунок не використовує бази даних, не зберігає стан користувачів чи "історію конфігурацій" між сесіями. Усі конфігурації генеруються "на льоту".

### Оновлення конфігурацій
- Якщо нова версія містить зміни у зовнішніх статичних файлах (модифіковані шаблони, налаштування портів запуску тощо), замініть ці конфігурації. У переважній більшості випадків Eel-застосунок збирається одним файлом, тому цей крок може бути необов'язковим.

## 3. Перевірка після оновлення
*(Примітка: Детальні тести правильності роботи, моніторинг продуктивності та можливі проблеми є темою окремих лабораторних, тому тут пропускаються).*

- **Базова перевірка (Smoke Test):** Запустіть оновлений `CiscoRouterWizard.exe` під тестовим акаунтом і переконайтеся, що GUI завантажується без критичних помилок (White Screen / Console Errors) і коректно відкриває стартову сторінку.
- Переконайтеся, що версія програми в UI (якщо така виводиться) відповідає очікуваному оновленню.

---

## Процедура відкату (Rollback)

У разі невдалого оновлення (наприклад, застосунок не запускається у користувачів, з'являються блокуючі помилки WebView, або не працює генератор конфігурацій) необхідно швидко виконати Rollback:

1. **Зупинка**: Одразу сповістіть користувачів про технічні труднощі та примусово завершіть усі процеси оновленої програми (`Stop-Process -Name "CiscoRouterWizard" -Force`).
2. **Видалення**: Видаліть скомпрометований файл оновлення з робочої папки.
3. **Відновлення з бекапу**: Поверніть на місце попередньо збережену робочу версію (перейменуйте назад у `CiscoRouterWizard.exe` з `_backup.exe`). Відновіть зовнішні конфігурації, якщо вони змінювалися.
4. **Перевірка відкату**: Запустіть відновлений додаток і переконайтеся, що базова працездатність повернулася до стабільного стану.
5. **Інцидент-менеджмент**: Зафіксуйте деталі збою у корпоративному баг-трекері та передайте команді розробників логи для аналізу.
