# Процедура резервного копіювання та відновлення (Backup & Restore Strategy)

Цей документ визначає політику, стратегію резервного копіювання та відновлення (Disaster Recovery) для проекту "Cisco Router Wizard". Він орієнтований на Release Engineer та DevOps.

> **Важливо:** Оскільки додаток є клієнтським генератором конфігурацій (stateless), він **не використовує базу даних** і не зберігає стан користувачів між сесіями на сервері. Це суттєво спрощує стратегію бекапів, зводячи її до резервного копіювання вихідного коду, зовнішніх конфігурацій (якщо такі змінені) та готових артефактів збірки.

## 1. Стратегія резервного копіювання

### Типи резервних копій
- **Повні (Full Backup):** Збереження всього репозиторію коду, документації, а також готового скомпільованого реліз-файлу (`.exe`). Також включає повний бекап директорії `backend/templates/`.
- **Інкрементальні/Диференціальні:** Реалізуються природним шляхом через систему контролю версій (Git). Кожен новий комміт є інкрементальною зміною до коду або статичних шаблонів.

### Частота створення резервних копій
- **Код та конфігурації (Git):** Після кожної зміни у гілці розробки. Віддалений Git-сервер (GitLab/GitHub) забезпечує безперервне збереження історії.
- **Релізні збірки (Artifacts):** Під час кожного планового релізу еталонний файл збірки копіюється у сховище. 
- **Термінальні сервери (Terminal Server):** Якщо застосунок працює у RDS середовищі, рекомендується налаштувати створення планових Snapshot-ів віртуальної машини 1 раз на тиждень (Weekly Differential).

### Зберігання та ротація копій
- Зберігання релізних файлів (`CiscoRouterWizard.exe`): У корпоративному сховищі артефактів (наприклад, Nexus, Artifactory, файловий SMB сервер або Release Assets у Git).
- Ротація (Retention Policy): Зберігати останні 5-10 Minor/Patch збірок. Старіші Patch-збірки видаляються. Мажорні версії (v1.0.0, v2.0.0) зберігаються безстроково.

## 2. Процедура резервного копіювання

### Бази даних
- **Додаток не має БД.** Проект не використовує РСУБД або NoSQL. Відповідно процедура SQL dump пропускається.

### Файли конфігурацій
- Шаблони Cisco IOS (.j2) лежать у `backend/templates/`. Їх резервне копіювання є частиною регулярного Git-флоу розробників. У виробничому середовищі ці шаблони або запаковані в `.exe`, або знаходяться поруч. В останньому випадку скрипт-бекапер має копіювати директорію шаблонів.

### Користувацькі дані
- **Не зберігаються.** Згенеровані IOS-конфігурації клієнт вручну копіює у буфер обміну. Дані користувачів не кешуються сервером і бекапу не підлягають.

### Логи системи
- Додаток може друкувати trace-логи (у Dev-режимі). Якщо у Production реалізовано логування у файл (наприклад, у `%APPDATA%\CiscoRouterWizard\logs`), вони збираються системами типу ELK Stack / Splunk. Резервне копіювання самих txt-логів з локальних дисків не є критичним для відновлення системи.

## 3. Перевірка цілісності та Автоматизація

- **Перевірка цілісності резервних копій:** Усі вихідні коди захищені механізмом хешування Git (SHA-1). Скомпільовані Production файли (`.exe`) перевіряються шляхом регулярного порівняння контрольних сум (SHA-256) з еталонними значеннями на Build-сервері, щоб уникнути пошкоджень під час передачі чи зберігання.
- **Автоматизація (скрипти, інструменти):** Процес бекапу коду повністю покладено на інфраструктуру Git. Створення релізних артефактів має бути автоматизовано через CI/CD (Pipeline Config), який викликає команду збірки та деплоїть отриманий артефакт до Remote Registry або сховища. Надані батники (`start_prod.bat`) спрощують локальний запуск і перевірку.

## 4. Процедура відновлення з резервних копій (Restore)

Оскільки динамічних даних та баз даних у системи немає, підходи до відновлення дуже надійні та швидкі. Отже:

### Повне відновлення системи
У разі фатального збою (видалення застосунку, зараження сервера):
1. **Ізоляція:** Видалити всі поточні скомпрометовані файли та завершити RDP-сесії.
2. **Завантаження з Artifact Repository:** Завантажити цілий, протестований файл `CiscoRouterWizard.exe` останньої стабільної версії зі сховища артефактів.
3. **Розгортання:** Розмістити файл у цільовій директорії налаштувань Production сервера. Додаток готовий до роботи одразу.

### Вибіркове відновлення даних
- Якщо було випадково видалено лише структуру зовнішніх конфігурацій або шаблонів (якщо вони використовуються як розпаковані файли у Prod-середовищі) — виконайте `git checkout main -- backend/templates/` і скопіюйте чисті файли у Prod-папку.

### Тестування відновлення (DR Testing)
- Запустіть "свіжовідновлений" `.exe` файл.
- У графічному інтерфейсі заповніть звичайний Hostname і перейдіть до генерації.
- Якщо логіка Jinja зібрала конфігурації без помилок, згенерований текст відображається і його можна копіювати, процес Disaster Recovery вважається 100% успішним.
